# Profile Tools: Collapse Frames, Remove Frames, Trim to Time Range

## Context

Jeffrey's profile Tools section currently has only "Rename Frames". Users need more data transformation tools to simplify flamegraph analysis, remove noise, and focus on specific time periods. This spec covers 3 new tools as the first batch of 8 planned tools.

**Target users:** Performance engineers, application developers, and security/compliance teams.

## Tools Overview

| Tool | Sidebar Section | Input | Database Target |
|------|----------------|-------|-----------------|
| Collapse Frames | TRANSFORM | pattern + label | stacktraces + frames |
| Remove Frames | TRANSFORM | pattern | stacktraces + frames |
| Trim to Time Range | FILTER | time range (brush chart) | events + cascade cleanup |

All tools follow the existing **Preview → Apply** pattern established by Rename Frames. All mutations clear the profile cache via `ProfileCacheRepository.clearAll()`.

## Tool 1: Collapse Frames

### Purpose
Replace consecutive stacktrace frames matching a class name pattern with a single synthetic frame. E.g., collapse all `org.springframework.*` frames into `[Spring Framework]`. Simplifies flamegraphs by hiding framework internals while preserving call structure.

### User Flow
1. Enter class name pattern (e.g., `org.springframework`)
2. Enter synthetic frame label (e.g., `[Spring Framework]`)
3. Click Preview → see matching frame count, affected stacktrace count, sample matching frames
4. Click Apply Collapse → confirmation dialog (permanent operation warning)
5. Confirm → operation executes, success message with stats

### REST API
```
POST /tools/collapse-frames/preview
  Request:  { "pattern": "org.springframework", "label": "[Spring Framework]" }
  Response: { "matchingFrames": 47, "affectedStacktraces": 1234, "samples": [{"className": "...", "methodName": "..."}] }

POST /tools/collapse-frames
  Request:  { "pattern": "org.springframework", "label": "[Spring Framework]" }
  Response: { "affectedStacktraces": 1234, "mergedStacktraces": 56 }
```

### Database Operations (Java-assisted)

The collapse operation requires array manipulation and hash recomputation, done in Java:

1. **Find matching frames**: `SELECT frame_hash FROM frames WHERE class_name LIKE '%' || :pattern || '%'`
2. **Create synthetic frame**: Insert into `frames` table with label as class_name, empty method_name, frame_type `Interpreted`
3. **Read affected stacktraces**: Query all stacktraces containing any matching frame hash
4. **Java processing**: For each affected stacktrace:
   - Replace consecutive runs of matching frame hashes with the synthetic frame hash
   - Recompute `stacktrace_hash` using `SingleThreadHasher` (XXHash64 — must match existing algorithm)
   - If multiple old stacktraces collapse to the same new hash → merge (keep one, redirect events)
5. **Write back**: Insert new stacktraces, update events FKs, delete old stacktraces
6. **Cleanup**: Delete orphaned frames not referenced by any stacktrace
7. **Clear cache**

### Why Java, Not Pure SQL
`stacktrace_hash` values were computed using `SingleThreadHasher` (XXHash64 over little-endian byte arrays). After modifying `frame_hashes` arrays, hashes must be recomputed with the same algorithm. DuckDB's `hash()` function would produce incompatible values. Processing in Java ensures hash consistency.

## Tool 2: Remove Frames

### Purpose
Strip frames matching a class name pattern from all stacktraces. Removes AOP proxies, lambda wrappers, generated bytecode, or framework noise.

### User Flow
1. Enter class name pattern (e.g., `$$Lambda$`)
2. Click Preview → see matching frame count, affected stacktrace count, sample matching frames
3. Warning shown if any stacktraces would become empty (all frames removed → events deleted)
4. Click Apply Remove → confirmation dialog
5. Confirm → operation executes

### REST API
```
POST /tools/remove-frames/preview
  Request:  { "pattern": "$$Lambda$" }
  Response: { "matchingFrames": 89, "affectedStacktraces": 2567, "samples": [{"className": "...", "methodName": "..."}] }

POST /tools/remove-frames
  Request:  { "pattern": "$$Lambda$" }
  Response: { "affectedStacktraces": 2567, "mergedStacktraces": 123, "removedEmptyStacktraces": 5 }
```

### Database Operations (Java-assisted)

Same pattern as Collapse Frames, but simpler:

1. **Find matching frames**: Same query as Collapse
2. **Read affected stacktraces**: Same query as Collapse
3. **Java processing**: For each affected stacktrace:
   - Remove all matching frame hashes from the array
   - If array becomes empty → mark stacktrace for deletion (events referencing it also deleted)
   - Otherwise: recompute `stacktrace_hash` using `SingleThreadHasher`
   - Handle hash collisions (multiple old stacktraces reduce to same new hash → merge)
4. **Write back**: Insert new stacktraces, update events FKs, delete old stacktraces, delete events with empty stacktraces
5. **Cleanup**: Delete orphaned frames, threads
6. **Clear cache**

### Shared Infrastructure with Collapse Frames

Both tools share:
- Query to find matching frame hashes by class name pattern
- Query to find affected stacktraces
- Batch operations: insert new stacktraces, update event FKs, delete old stacktraces
- Orphan cleanup: delete unreferenced frames, stacktraces, threads
- Hash recomputation using `SingleThreadHasher`

This shared logic lives in a common repository or utility, avoiding duplication.

## Tool 3: Trim to Time Range

### Purpose
Permanently remove events outside a selected time window. Uses the `TimeSeriesChart` brush component (same as Flamegraph view) for intuitive range selection.

### User Flow
1. Page loads timeseries data for the profile's primary event type (e.g., `jdk.ExecutionSample`)
2. `TimeSeriesChart` renders with brush chart showing the full profile duration
3. User drag-selects the range to **keep** on the brush chart
4. Selection summary appears: events to keep, events to remove, selected range
5. Click Trim Profile → confirmation dialog
6. Confirm → operation executes, success message with cleanup stats

### REST API
```
POST /tools/trim-time-range/preview
  Request:  { "start": "2026-03-28T14:30:00Z", "end": "2026-03-28T14:35:00Z" }
  Response: { "totalEvents": 45678, "eventsToRemove": 32100, "eventsToKeep": 13578, "currentStart": "...", "currentEnd": "..." }

POST /tools/trim-time-range
  Request:  { "start": "2026-03-28T14:30:00Z", "end": "2026-03-28T14:35:00Z" }
  Response: { "removedEvents": 32100, "removedStacktraces": 890, "removedFrames": 234, "removedThreads": 12 }
```

### Database Operations (Pure SQL)

No array manipulation needed — straightforward deletions:

```sql
-- Step 1: Delete events outside range
DELETE FROM events WHERE start_timestamp < :start OR start_timestamp > :end;

-- Step 2: Delete orphaned stacktraces
DELETE FROM stacktraces WHERE stacktrace_hash NOT IN (
    SELECT DISTINCT stacktrace_hash FROM events WHERE stacktrace_hash IS NOT NULL
);

-- Step 3: Delete orphaned frames
DELETE FROM frames WHERE frame_hash NOT IN (
    SELECT DISTINCT unnest(frame_hashes) FROM stacktraces
);

-- Step 4: Delete orphaned threads
DELETE FROM threads WHERE thread_hash NOT IN (
    SELECT DISTINCT thread_hash FROM events WHERE thread_hash IS NOT NULL
);
```

### Frontend: TimeSeriesChart Reuse

The Vue component uses `TimeSeriesChart` with:
- `zoomEnabled: true` to enable brush selection
- Listens to `@update:timeRange` for selection events
- Loads timeseries data via the existing timeseries API
- Displays selection summary below the chart
- No `graphUpdater` prop — uses the emit-based communication path

## Module Structure

New Maven modules under `jeffrey-local/profiles/tools/`:

```
jeffrey-local/profiles/tools/
├── pom.xml                    (parent POM)
├── collapse-frames/
│   ├── pom.xml
│   └── src/main/java/pbouda/jeffrey/profile/tools/collapse/
│       ├── CollapseFramesManager.java          (interface)
│       ├── CollapseFramesManagerImpl.java       (implementation)
│       └── CollapseFramesResource.java          (JAX-RS resource)
├── remove-frames/
│   ├── pom.xml
│   └── src/main/java/pbouda/jeffrey/profile/tools/remove/
│       ├── RemoveFramesManager.java
│       ├── RemoveFramesManagerImpl.java
│       └── RemoveFramesResource.java
└── trim-time-range/
    ├── pom.xml
    └── src/main/java/pbouda/jeffrey/profile/tools/trim/
        ├── TrimTimeRangeManager.java
        ├── TrimTimeRangeManagerImpl.java
        └── TrimTimeRangeResource.java
```

Each module contains its own manager interface/impl and JAX-RS resource. Shared repository operations (stacktrace transformation, orphan cleanup) are added to the persistence layer (`profile-persistence-api` / `profile-sql-persistence`).

### Repository Layer

New repository interface in `profile-persistence-api`:

**`ProfileToolsRepository`** — shared operations for stacktrace-modifying tools:
- `findFrameHashesByClassNamePattern(pattern)` → `List<Long>`
- `findStacktracesContainingFrames(frameHashes)` → `List<StacktraceRecord>`
- `insertSyntheticFrame(hash, className, methodName)`
- `applyStacktraceTransformation(hashMapping, newStacktraces)` — batch update events FK, insert new stacktraces, delete old
- `deleteOrphanedFrames()`, `deleteOrphanedStacktraces()`, `deleteOrphanedThreads()`
- `previewTrimTimeRange(start, end)` → `TrimPreview`
- `trimTimeRange(start, end)` → counts of deleted rows per table

JDBC implementation in `profile-sql-persistence`: **`JdbcProfileToolsRepository`**

### Configuration

Each tool module provides its own `@Configuration` class with a `Manager.Factory` bean. The factory creates manager instances per profile using `DatabaseManagerResolver` to open the profile's DuckDB.

The resources are registered in the existing `ProfileResource` sub-resource wiring (or via a new tools sub-resource locator).

1. **Infrastructure**: `ProfileToolsRepository` interface + `JdbcProfileToolsRepository` with shared query methods
2. **Trim to Time Range**: Simplest tool (pure SQL, no array manipulation) — proves the module structure and frontend pattern

## Verification

1. **Collapse Frames**: Create a profile with Spring Boot stacktraces. Collapse `org.springframework` into `[Spring Framework]`. Verify flamegraph shows synthetic frame, timeseries still works, guardian analysis still runs.
2. **Remove Frames**: Remove `$$Lambda$` frames. Verify stacktraces are shorter in flamegraph. Check that event counts remain correct (unless stacktraces became empty).
3. **Trim to Time Range**: Record a 2-minute profile. Trim to 30s–60s window. Verify all views only show data from that window. Check that timeseries, flamegraph, and thread views reflect the trimmed range.
4. **Cache invalidation**: After each tool operation, verify that flamegraph and timeseries views show updated data without manual refresh.
5. **Frontend**: Run `npm run lint` and `npm run test` in `pages-local/`.
6. **Backend**: Run `mvn compile` to verify Java compilation.
